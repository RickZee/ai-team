## TL;DR

The pipeline's failures on the LangGraph backend are **agentic, not code** failures: the
open-weights dev/QA model (deepseek-v3 via OpenRouter) behaves like a **chatbot, not an
autonomous agent** — it writes code as prose and asks "shall I save this?" instead of
calling the `file_writer` tool. The orchestration then loops: no file on disk → QA fails
→ retry → same prose → repeat, until retries exhaust. The work this session was to make
that failure **bounded, cheap, observable, and partially self-healing**, and to give all
three backends the same spend/loop protection.

Result: runs that used to hang for 20+ min or burn money in a tight loop now abort in
seconds with a clear reason, salvage what the model wrote, and fail clean.

## The core agentic problem: prose instead of tool-calls

This is the through-line for almost everything below. Worth understanding deeply.

### What the model does

The LangGraph subgraphs use ReAct agents (`create_react_agent`) given a `file_writer`
tool and instructions like *"Call file_writer — do not output test code as plain text."*
deepseek-v3 **ignores this** and instead emits, as ordinary assistant text:

> "Here are the files:
> ### 1. `main.py` (Flask Application)
> ```python
> from flask import Flask
> ...
> ```
> Would you like me to proceed with saving these files?"

It produces **correct code** — but as markdown prose, framed as a *proposal awaiting
approval*, never as a tool call. There is no human to approve, so nothing is written.
Observed at every phase: planning supervisor, development supervisor, QA engineer.

### Why this breaks the pipeline (the loop)

The phases coordinate through **files on disk**, not the chat transcript (per CLAUDE.md:
file-based handoff). So when dev writes `main.py` as prose:

1. Development "completes" — but workspace has no app module.
2. Testing runs the real quality gate (`pytest`) → `ImportError: no module named main`
   or `exit 5: no tests collected`.
3. `route_after_testing` sees the failure → routes to `retry_development`.
4. Development re-runs, emits the same prose again, still no tool call.
5. Loop until `max_retries` (3) → escalate / fail.

Each turn of the loop is a **full dev+testing phase** = several LLM calls on a slow model.
Pre-guardrail, this looked like a 5–20 min hang that produced nothing.

### Secondary agentic quirks (same model, OpenRouter)

- **Malformed tool-call args.** When deepseek *does* attempt a tool call, it sometimes
  serializes `function.arguments` as a parsed dict instead of a JSON string, or emits
  truncated/prose-wrapped JSON. This crashed the subgraph mid-run with a `JSONDecodeError`
  ("line 525 column 1") — a hard failure masquerading as a content problem.
- **Empty LLM responses.** OpenRouter occasionally returns `None`/empty for a completion
  (seen with crewai last session as `Invalid response from LLM - None or empty`). Looks
  identical to a real failure; must be retryable, not fatal.
- **Self-narration без action.** The QA agent literally logs *"I will now proceed to write
  the test files… Let me first inspect the source files. (;"* and then stops — a planning
  monologue with no tool invocation.

### Why this is *not* just "use a better model"

True — Claude models call tools reliably and the Claude Agent SDK backend doesn't show
this. But the orchestration must be **robust to non-compliant models** because:
- The whole point of the project is multi-backend comparison incl. open models.
- Even strong models occasionally narrate instead of acting.
- A pipeline that silently loops/burns money on bad model behavior is unsafe regardless of
  which model is default.

So the fixes are defensive guardrails, not a model swap.

## Solutions shipped (G1–G7 + crewai budget)

Ordered by how directly they address the agentic failure.

### Salvage: recover prose-as-files (G3, G6, G7) — the heart of it

If the model writes code as fenced markdown with a filename, **extract it and write it to
disk ourselves**, so the loop can make progress despite the model never calling
`file_writer`.

- **G3** — added salvage to the *testing* node + a markdown-header regex
  (``### `test_calc.py` ``). Rescued Scenario 1 (smoke test), which now passes.
- **G6** — widened the regex: the dev supervisor uses *numbered, suffixed* headers like
  ``### 1. `main.py` (Flask Application)`` and ``### 2. Updated `tests/test_api.py` `` that
  the strict pattern missed (filename not immediately after the `#`s). New
  `_MD_HEADER_LOOSE_RE` matches a **backtick-wrapped** filename anywhere in a header line
  (backticks required, to avoid capturing prose that merely mentions a file).
- **G7** — the dev-phase salvage was gated on `if not generated` (workspace empty), so once
  *any* file existed (a test from a prior retry, a small dev write) it **skipped** —
  silently dropping the prose-written `main.py`. This was the actual loop driver. Now the
  dev node **always** runs extraction over its message delta; the extractor is idempotent
  and overwrites with the latest prose (a retry's corrected version wins).

Effect: across iterations, the todo-app run went from "1 test file, nothing else" to
landing `todoapp/app.py`, `requirements.txt`, packaged tests — the app code now reaches
disk. (See "Open problem" — it still doesn't fully pass within retries.)

> **Security note baked into salvage:** the extractor validates filenames *before* writing.
> A pre-existing `lstrip("./")` was stripping arbitrary leading dot/slash chars, turning
> `../escape.py` into `escape.py` and defeating the traversal check — a real path-traversal
> hole, now fixed (strip at most one `./`, reject absolute paths, `..` segments, dotfiles).
> Adversarial tests cover it. **Do not reintroduce `lstrip("./")`.**

### Recoverable routing + error reset (G2)

A *testing-phase* error (prose-not-tool-call crash, malformed tool-call) used to route
straight to terminal `error` — one bad model turn killed the whole run. Now
`route_after_testing` treats a testing-phase error as **retryable** (back to development up
to `max_retries`, then escalate to human). Errors from other phases stay terminal.

This exposed a latent bug: the `errors` state field used `operator.add`, so nodes returning
`"errors": []` to signal "recovered" were a **no-op append** — stale errors persisted and
forced terminal routing even after recovery. New `reset_or_extend_errors` reducer treats an
empty update as a reset. `retry_development` now clears errors each attempt.

Also: `pytest` exit 5 ("no tests collected") is surfaced as a distinct
`no_tests_collected` signal with actionable reason text, so a retry can tell the agent
"you wrote no tests" rather than treating it like an ordinary test failure.

### Bounding cost and loops (G4, crewai budget)

The retry loop above is *bounded by count* (`max_retries`, `MAX_SUBGRAPH_GUARDRAIL_RETRIES`,
recursion limit) but nothing bounded **spend**. A crash/retry loop on a slow model burns
real dollars.

- **G4 (LangGraph)** — a per-run spend ceiling fed by the OpenRouter `usage.cost` in the
  httpx response hook; explicit `recursion_limit=50` instead of the implicit 25.
- **crewai budget (this session)** — CrewAI had only an *error-count* budget
  (`MAX_RUN_ERRORS=50`), not a dollar ceiling. Added a LiteLLM success callback feeding the
  same shared guard (`ai_team.core.spend_guard`), so all three backends now enforce a
  dollar ceiling (Claude SDK already had `max_budget_usd`).

**Key design — non-retryable by construction:** `BudgetExceededError` subclasses
**`BaseException`**, not `Exception`. This is deliberate and load-bearing:
- LangGraph phase nodes wrap `sub.invoke` in `except Exception` and convert failures into
  *retryable* error dicts. A budget abort must not be retried (retrying is what we're
  stopping).
- LiteLLM wraps success callbacks in `except Exception` and treats failures as
  "non-blocking" (logs and continues).

By subclassing `BaseException`, the abort **bypasses both** and propagates out to fail the
run cleanly. Caught explicitly only at the backend run/stream boundary → `success=False`,
exit 1. Verified end-to-end: `AI_TEAM_RUN_BUDGET_USD=0.002` aborts at the 3rd LLM call.

### Wall-clock watchdog (G1)

`run_demo.py` had no internal timeout — a hung LLM/tool call blocked forever (saw a 22-min
silent hang, and last session a 31-min one). Added `--timeout` (default 900s) arming a
`SIGALRM` watchdog → abort with exit 124. This is the CLI-level backstop above all the
in-graph guards.

### Spend-hook coverage (G5)

The spend *tracker* was tested, but the **hook that feeds it** (`langgraph_chat.py`,
parsing OpenRouter's `usage.cost`) had zero tests. If OpenRouter's response shape drifts,
the guard would silently record \$0 and never fire — a money-safety blind spot. Added 12
tests over the real wire shapes (cost top-level / nested / missing / malformed).

## Open problems / next steps (agentic focus)

### 1. The dev supervisor still under-produces even when salvaged

Latest Scenario 2 run landed `app.py` + `requirements.txt` but the QA test file imports a
module path that doesn't match the app the dev wrote (`from main import app` vs
`todoapp/app.py`). The agents don't agree on a layout because **no agent owns the project
structure** — each phase independently invents paths. The salvage gets files onto disk but
can't reconcile cross-file contracts.
→ Consider a planning-phase artifact that pins the file layout, injected into dev + QA
prompts, so imports line up. This is an *agent-coordination* fix, not a regex fix.

### 2. Salvage is a band-aid for a prompting problem

We're recovering from the model not calling `file_writer`. Better upstream: a stronger
system prompt / few-shot that forces tool use, or a `tool_choice="required"`-style
constraint where the provider supports it. Salvage should remain as defense-in-depth, but
reducing prose-output at the source would cut the retry waste (and cost/latency) entirely.

### 3. Re-prompt on detected prose-not-tool-call

Today we salvage *after* the fact. A tighter loop: detect "agent emitted code blocks but
made no `file_writer` call" within the ReAct loop and **re-prompt immediately** ("you must
call file_writer"), before the phase ends. Cheaper than a full phase retry.

### 4. Latency / cost of the retry loop

Even bounded, 3 retries × multi-call phases × ~30–60s/call (deepseek) = several wasted
minutes per failed run. The salvage + re-prompt ideas above attack this; a faster default
model for demos would too.

### 5. Remaining test/coverage gaps (lower priority, not agentic)

- **Gap 4 — coverage gate:** added `fail_under = 60` to `[tool.coverage.report]` in
  pyproject (conservative floor; CI already runs `--cov`). **Not yet committed / verified**
  — full-suite coverage couldn't be measured locally (OOM under memory pressure). Verify the
  real number in CI and ratchet the floor up.
- **Gap 3 — claude-sdk untested modules:** `reasoning_log.py`, `workspace_snapshots.py`
  (file rollback — safety-relevant), `streaming.py` have **0 test references**;
  `recovery.py` is thin. Not started.

## Quick reference

```bash
# Run a demo with all guardrails + a dollar budget + wall-clock cap
AI_TEAM_RUN_BUDGET_USD=3.00 uv run python scripts/run_demo.py \
  demos/02_todo_app --backend langgraph --skip-estimate --timeout 900

# Env knobs
AI_TEAM_RUN_BUDGET_USD            # per-run spend ceiling (USD); 0 disables. default 5
AI_TEAM_LANGGRAPH_RECURSION_LIMIT # graph superstep cap. default 50
# --timeout (CLI)                 # wall-clock seconds; 0 disables. default 900

# Tail a backgrounded run's real log (harness temp files get reaped)

grep -iE "fallback_extraction|spend_guard|current_phase|success" <logfile>
```
