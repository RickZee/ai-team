## New problems found + solutions (this session)

### 1. Web runs died on tab navigation (`/ws/run` socket-bound lifetime)

**Symptom:** launching a run via the web UI, then navigating away from the Run tab
(which the UI does automatically after `run_started`), killed the run in ~3s with an
empty error string and an SDK-side `Stream closed` in stderr.

**Root cause:** `_execute_run` streamed events over the `/ws/run` WebSocket and was
literally bound to that socket's lifetime. `useRunWebSocket`'s cleanup effect closes the
socket on unmount (Run→Dashboard navigation unmounts it); the next `ws.send_json` raised,
`ws_run` caught it and cancelled the task, which tore down the SDK subprocess's control
stream. claude-agent-sdk hit this **every time** because its first phase alone takes
minutes — the socket is guaranteed to be closed before any real work lands.

**Fix** (`src/ai_team/ui/web/server.py`):
- `_safe_send()` — swallow send failures to a dead client instead of propagating them.
- `_spawn_detached_run()` + a strong-ref set (`_DETACHED_RUNS`) — the run task now
  survives the request scope; only an explicit `state.cancel_run()` cancels it.
- Client disconnect no longer cancels the run; the Dashboard's separate
  `/ws/monitor/{run_id}` socket observes it going forward.

**Verified live**: ran claude-agent-sdk through the browser, navigated away immediately —
run reached `complete` (7m27s, $1.09, all phases green) instead of dying at 3s. Plus 2
regression tests (`tests/unit/ui/test_web_websocket.py`) — note TestClient's
in-process portal kills *all* loop tasks on teardown, so the test asserts the
cancel-flag contract (disconnect ≠ cancel-request), not literal task survival.

### 2. Runtime smoke gate + self-improvement loop (the "tests pass, app doesn't work" gap)

**Symptom, discovered while showcasing the todo app:** claude-agent-sdk headless run
reported `success: true`, **70/70 pytest passed**, full app generated (Flask + SQLite +
Docker) — but the actual running app **500'd on every request** (`GET /health -> 500`,
`structlog.stdlib.add_logger_name` paired with `PrintLoggerFactory`, which has no
`.name`). Unit tests used an in-process Flask test client that never exercises the real
logging chain, so "tests pass" silently meant nothing about runtime correctness. No
agent, in any backend, ever *ran* the generated app.

**Fix — shared gate, backend-specific loops** (design in `docs/SELF_IMPROVEMENT.md`):

- **`ai_team.tools.smoke_tools.run_app_smoke(workspace)`** (new) — boots the app
  (`docker compose up -d --build`, else a Flask `module:app`/`create_app()` under `src/`
  via the stdlib server), probes real HTTP, writes `docs/smoke_results.json`. Supports a
  **stateful probe sequence** from `expected_output.json`'s `smoke` list: `save` captures
  a JSON response field (e.g. an id from a `POST`) and `{var}` tokens interpolate it into
  later `path`/`body`, so a contract can drive create→read→update→delete, not just a
  lone health check. `shell=False`/arg-lists throughout.
  - `load_or_run_smoke(workspace, max_age_s=600)` reuses a fresh cached result instead
    of re-booting per consumer (agent MCP call, recovery loop, post-run gate) — avoids
    repeated `docker compose up --build`/`down` cycles.
  - Skips cleanly (`ran=false`, not a failure) when there's no bootable entrypoint or
    Docker is unavailable; skips a compose boot if the publish port is already bound by
    something else (avoids a false pass/fail probing a foreign service).
- **`runtime_smoke_guardrail`** (`guardrails/quality.py`) — fails when `ran && !success`.
- **`main.py`** post-run gate now runs the smoke check for **every** backend (booting
  the app itself if the backend's own agents didn't), so LangGraph/CrewAI get the same
  coverage as the SDK agents get natively.
- **claude-agent-sdk loop**: `run_orchestrator_with_recovery` applies the smoke gate
  between attempts, feeds `{endpoint, status, traceback, server logs}` back as a fix
  prompt, retries (`settings.anthropic.smoke_loop_enabled` default on,
  `smoke_max_attempts` default 2). QA + DevOps agents also got the `run_app_smoke` MCP
  tool to self-correct mid-run.
- **LangGraph loop**: new `smoke` node between `testing` and `deployment`;
  `route_after_testing` now sends a passing suite to `smoke` (not straight to
  `deployment`); `route_after_smoke` retries via the existing `retry_development` edge
  (bounded by `retry_count`/`max_retries`) on failure, escalates to `human_review` when
  exhausted. Placeholder-mode graph records a clean skip so unit tests stay
  deterministic; full-mode calls the real gate.
- **CrewAI**: shared post-run gate only, no inner loop (matches its comparison-only
  status — not worth an inner loop for a backend already known-red).

**Verified end-to-end**: ran the gate against the actual broken to-do app —
`GET /health -> 500` caught, exactly the defect 70 green pytests missed. Live
create→read→delete round-trip against a real Flask app (id captured from `POST`,
threaded into `GET`/`DELETE`). Drove the real LangGraph graph with a fail-then-pass
smoke mock: looped through `development` once, then reached `complete` with
`retry_count` incremented. **494 tests pass** across the affected suites (tools,
guardrails, backends, ui), ruff + mypy clean.

Code-reviewed (own `/code-review`) and fixed 4 findings from that pass: compose
foreign-port collision, redundant re-boots, `_drain_logs` blocking on a partial log
line (`readline()` → non-blocking `os.read`), and the stateless-probe limitation above.

### 3. Web UI silently ran LangGraph in `placeholder` mode (new regression, this session)

**Symptom:** the first Compare-tab attempt showed LangGraph's column go from "Starting"
to `COMPLETE` in **7 seconds** with 0 files, 0 tasks, 0 cost. Looked like an instant
crash; was actually an instant *fake* success.

**Root cause:** `LangGraphBackend._graph_mode()` defaults to `"placeholder"` (a no-LLM
stub graph, used for fast unit tests) unless `graph_mode="full"` is passed explicitly or
`AI_TEAM_LANGGRAPH_GRAPH_MODE` is set. The CLI (`scripts/run_demo.py`) already defaults
to `graph_mode="full"` — but **`src/ai_team/ui/web/server.py`'s** `_execute_run` /
`_stream_langgraph_events_to_ws` never passed it at all. Every LangGraph run launched
through the web UI (Run tab or Compare tab) was silently running the placeholder stub,
not a real pipeline — this predates this session and would have quietly invalidated any
prior web-UI-driven LangGraph comparison.

**Fix**: both the streaming dispatch and the HITL `resume()` call now default to
`graph_mode=os.environ.get("AI_TEAM_LANGGRAPH_GRAPH_MODE", "full")`, matching the CLI's
intent — web runs are real runs, not test scaffolding.

**Verified**: re-ran the same Compare brief; log showed `langgraph_main_graph_compiled
mode=full`, and the run progressed through real phases (`development` at 58s, not an
instant complete).

### 4. CrewAI `Task` guardrail return-type validation crash (new, blocks *both* CrewAI and LangGraph's dev phase)

**Symptom, surfaced only after fixing #3** (placeholder mode had been silently hiding
this — the stub graph never calls into the real crew code): once LangGraph actually ran
`mode=full`, its development phase crashed:

```
1 validation error for Task
  Value error, If return type is annotated, it must be Tuple[bool, Any]
    input_value=<function create_backend_...guardrail_fn>
flow=AITeamFlow
```

**Root cause:** `src/ai_team/tasks/development_tasks.py` and `testing_tasks.py` both had
`from __future__ import annotations` at module level, which makes every annotation in
the file a **lazy string** at runtime (PEP 563). CrewAI's `Task` pydantic validator
(`crewai/task.py`) does `inspect.signature(fn).return_annotation` and expects an actual
`tuple[bool, Any]` **generic alias** object (checks `get_origin(...) is tuple`); with the
future import it received the *string* `"tuple[bool, Any]"` instead, which fails the
`get_origin` check and raises. This is a shared-code bug: LangGraph's development
subgraph delegates to `crews/development_crew.py`'s guardrailed `Task`s, so the crash hit
LangGraph even though the crash *originates* in CrewAI's validator — explains why both
LangGraph's dev phase and (per 06-28) CrewAI's own runs have been unreliable around this
exact area.

**Fix**: removed `from __future__ import annotations` from both files. Safe — the only
annotations relying on it are PEP 604 `X | None` unions, which resolve natively at
runtime on this repo's minimum Python (3.11+); no forward references depend on lazy
evaluation.

**Verified**: reproduced the exact guardrail closure with a real agent (not a mock —
pydantic validates the type), confirmed `return_annotation` now resolves to the real
`tuple[bool, typing.Any]` generic alias and CrewAI's check passes. Also checked
`planning_tasks.py` (never had the future import, unaffected) and confirmed
`testing_tasks.py`'s guardrail functions were unannotated (never actually hit by this
bug, but fixed the file for consistency). 13 relevant tests pass, ruff/mypy clean.

**Not yet re-verified end-to-end** — the retry comparison run that would prove this fix
lets LangGraph's dev phase complete was interrupted by the dev servers dying mid-run
(see Known issues below) before reaching a conclusive result.
