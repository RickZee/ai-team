## New problems found + solutions (this session)

### 1. Web runs died on tab navigation (`/ws/run` socket-bound lifetime)

**Symptom:** launching a run via the web UI, then navigating away from the Run tab
(which the UI does automatically after `run_started`), killed the run in ~3s with an
empty error string and an SDK-side `Stream closed` in stderr.

**Root cause:** `_execute_run` streamed events over the `/ws/run` WebSocket and was
literally bound to that socket's lifetime. `useRunWebSocket`'s cleanup effect closes the
socket on unmount (Run→Dashboard navigation unmounts it); the next `ws.send_json` raised,
`ws_run` caught it and cancelled the task, which tore down the SDK subprocess's control
stream. claude-agent-sdk hit this **every time** because its first phase alone takes
minutes — the socket is guaranteed to be closed before any real work lands.

**Fix** (`src/ai_team/ui/web/server.py`):
- `_safe_send()` — swallow send failures to a dead client instead of propagating them.
- `_spawn_detached_run()` + a strong-ref set (`_DETACHED_RUNS`) — the run task now
  survives the request scope; only an explicit `state.cancel_run()` cancels it.
- Client disconnect no longer cancels the run; the Dashboard's separate
  `/ws/monitor/{run_id}` socket observes it going forward.

**Verified live**: ran claude-agent-sdk through the browser, navigated away immediately —
run reached `complete` (7m27s, $1.09, all phases green) instead of dying at 3s. Plus 2
regression tests (`tests/unit/ui/test_web_websocket.py`) — note TestClient's
in-process portal kills *all* loop tasks on teardown, so the test asserts the
cancel-flag contract (disconnect ≠ cancel-request), not literal task survival.

### 2. Runtime smoke gate + self-improvement loop (the "tests pass, app doesn't work" gap)

**Symptom, discovered while showcasing the todo app:** claude-agent-sdk headless run
reported `success: true`, **70/70 pytest passed**, full app generated (Flask + SQLite +
Docker) — but the actual running app **500'd on every request** (`GET /health -> 500`,
`structlog.stdlib.add_logger_name` paired with `PrintLoggerFactory`, which has no
`.name`). Unit tests used an in-process Flask test client that never exercises the real
logging chain, so "tests pass" silently meant nothing about runtime correctness. No
agent, in any backend, ever *ran* the generated app.

**Fix — shared gate, backend-specific loops** (design in `docs/SELF_IMPROVEMENT.md`):

- **`ai_team.tools.smoke_tools.run_app_smoke(workspace)`** (new) — boots the app
  (`docker compose up -d --build`, else a Flask `module:app`/`create_app()` under `src/`
  via the stdlib server), probes real HTTP, writes `docs/smoke_results.json`. Supports a
  **stateful probe sequence** from `expected_output.json`'s `smoke` list: `save` captures
  a JSON response field (e.g. an id from a `POST`) and `{var}` tokens interpolate it into
  later `path`/`body`, so a contract can drive create→read→update→delete, not just a
  lone health check. `shell=False`/arg-lists throughout.
  - `load_or_run_smoke(workspace, max_age_s=600)` reuses a fresh cached result instead
    of re-booting per consumer (agent MCP call, recovery loop, post-run gate) — avoids
    repeated `docker compose up --build`/`down` cycles.
  - Skips cleanly (`ran=false`, not a failure) when there's no bootable entrypoint or
    Docker is unavailable; skips a compose boot if the publish port is already bound by
    something else (avoids a false pass/fail probing a foreign service).
- **`runtime_smoke_guardrail`** (`guardrails/quality.py`) — fails when `ran && !success`.
- **`main.py`** post-run gate now runs the smoke check for **every** backend (booting
  the app itself if the backend's own agents didn't), so LangGraph/CrewAI get the same
  coverage as the SDK agents get natively.
- **claude-agent-sdk loop**: `run_orchestrator_with_recovery` applies the smoke gate
  between attempts, feeds `{endpoint, status, traceback, server logs}` back as a fix
  prompt, retries (`settings.anthropic.smoke_loop_enabled` default on,
  `smoke_max_attempts` default 2). QA + DevOps agents also got the `run_app_smoke` MCP
  tool to self-correct mid-run.
- **LangGraph loop**: new `smoke` node between `testing` and `deployment`;
  `route_after_testing` now sends a passing suite to `smoke` (not straight to
  `deployment`); `route_after_smoke` retries via the existing `retry_development` edge
  (bounded by `retry_count`/`max_retries`) on failure, escalates to `human_review` when
  exhausted. Placeholder-mode graph records a clean skip so unit tests stay
  deterministic; full-mode calls the real gate.
- **CrewAI**: shared post-run gate only, no inner loop (matches its comparison-only
  status — not worth an inner loop for a backend already known-red).

**Verified end-to-end**: ran the gate against the actual broken to-do app —
`GET /health -> 500` caught, exactly the defect 70 green pytests missed. Live
create→read→delete round-trip against a real Flask app (id captured from `POST`,
threaded into `GET`/`DELETE`). Drove the real LangGraph graph with a fail-then-pass
smoke mock: looped through `development` once, then reached `complete` with
`retry_count` incremented. **494 tests pass** across the affected suites (tools,
guardrails, backends, ui), ruff + mypy clean.

Code-reviewed (own `/code-review`) and fixed 4 findings from that pass: compose
foreign-port collision, redundant re-boots, `_drain_logs` blocking on a partial log
line (`readline()` → non-blocking `os.read`), and the stateless-probe limitation above.

### 3. Web UI silently ran LangGraph in `placeholder` mode (new regression, this session)

**Symptom:** the first Compare-tab attempt showed LangGraph's column go from "Starting"
to `COMPLETE` in **7 seconds** with 0 files, 0 tasks, 0 cost. Looked like an instant
crash; was actually an instant *fake* success.

**Root cause:** `LangGraphBackend._graph_mode()` defaults to `"placeholder"` (a no-LLM
stub graph, used for fast unit tests) unless `graph_mode="full"` is passed explicitly or
`AI_TEAM_LANGGRAPH_GRAPH_MODE` is set. The CLI (`scripts/run_demo.py`) already defaults
to `graph_mode="full"` — but **`src/ai_team/ui/web/server.py`'s** `_execute_run` /
`_stream_langgraph_events_to_ws` never passed it at all. Every LangGraph run launched
through the web UI (Run tab or Compare tab) was silently running the placeholder stub,
not a real pipeline — this predates this session and would have quietly invalidated any
prior web-UI-driven LangGraph comparison.

**Fix**: both the streaming dispatch and the HITL `resume()` call now default to
`graph_mode=os.environ.get("AI_TEAM_LANGGRAPH_GRAPH_MODE", "full")`, matching the CLI's
intent — web runs are real runs, not test scaffolding.

**Verified**: re-ran the same Compare brief; log showed `langgraph_main_graph_compiled
mode=full`, and the run progressed through real phases (`development` at 58s, not an
instant complete).

### 4. CrewAI `Task` guardrail return-type validation crash (new, blocks *both* CrewAI and LangGraph's dev phase)

**Symptom, surfaced only after fixing #3** (placeholder mode had been silently hiding
this — the stub graph never calls into the real crew code): once LangGraph actually ran
`mode=full`, its development phase crashed:

```
1 validation error for Task
  Value error, If return type is annotated, it must be Tuple[bool, Any]
    input_value=<function create_backend_...guardrail_fn>
flow=AITeamFlow
```

**Root cause:** `src/ai_team/tasks/development_tasks.py` and `testing_tasks.py` both had
`from __future__ import annotations` at module level, which makes every annotation in
the file a **lazy string** at runtime (PEP 563). CrewAI's `Task` pydantic validator
(`crewai/task.py`) does `inspect.signature(fn).return_annotation` and expects an actual
`tuple[bool, Any]` **generic alias** object (checks `get_origin(...) is tuple`); with the
future import it received the *string* `"tuple[bool, Any]"` instead, which fails the
`get_origin` check and raises. This is a shared-code bug: LangGraph's development
subgraph delegates to `crews/development_crew.py`'s guardrailed `Task`s, so the crash hit
LangGraph even though the crash *originates* in CrewAI's validator — explains why both
LangGraph's dev phase and (per 06-28) CrewAI's own runs have been unreliable around this
exact area.

**Fix**: removed `from __future__ import annotations` from both files. Safe — the only
annotations relying on it are PEP 604 `X | None` unions, which resolve natively at
runtime on this repo's minimum Python (3.11+); no forward references depend on lazy
evaluation.

**Verified**: reproduced the exact guardrail closure with a real agent (not a mock —
pydantic validates the type), confirmed `return_annotation` now resolves to the real
`tuple[bool, typing.Any]` generic alias and CrewAI's check passes. Also checked
`planning_tasks.py` (never had the future import, unaffected) and confirmed
`testing_tasks.py`'s guardrail functions were unannotated (never actually hit by this
bug, but fixed the file for consistency). 13 relevant tests pass, ruff/mypy clean.

**Verified end-to-end later in the session** (after switching to `nohup`-detached
servers, see "Dev servers kept dying" below) — see §5-§8 for what that verification
found and fixed next.

### 5. Scope-control guardrail false-positive on the planning supervisor

**Symptom:** once #3/#4 let LangGraph actually run `mode=full`, its **planning** phase
started failing 100% of the time: `"Output deviates from task scope (relevance 23-25%
below 50%)"`, 3 retries exhausted, hard error. A real, correct, on-scope architecture doc
(Flask/SQLite/Docker, matching the brief exactly) was being rejected.

**Root cause, two parts:**
1. `scope_control_guardrail`'s metric was `overlap = |requirement_words ∩ output_words| /
   |requirement_words|` — pure recall. An architecture doc that correctly *elaborates* the
   brief in its own vocabulary ("Repository", "ACID", "responsibilities" instead of
   repeating "Flask"/"SQLite"/"todos") scores low even though it's fully on-scope.
2. The check ran even on the **planning supervisor's** delegation/routing text ("assign
   requirements to product_owner, then architecture to architect") — pure meta-commentary
   with legitimately near-zero requirement-word overlap. No threshold can distinguish that
   from real off-topic drift.

**Fix** (`src/ai_team/guardrails/behavioral.py`, `langgraph_guardrail_nodes.py`):
`scope_control_guardrail` now takes `max(recall, precision)` — precision (how much of the
*output* is itself requirement-derived) lets a terse, correct elaboration through while
still failing genuinely off-topic output (both directions are low). Separately,
`_behavioral_stack` now skips scope-control entirely when `is_supervisor=True`;
`role_adherence_guardrail` still checks the supervisor stays in its delegation lane.

**Verified**: reproduced against the actual rejected `architecture.json` from the failing
run, plus 6 new regression tests. Re-run: LangGraph's planning phase now completes
cleanly (`confidence=1.0`) under real `mode=full` — the first time this session.

### 6. Web API misreported a genuine `human_review` escalation as `"complete"`

**Symptom:** LangGraph reached the testing phase, correctly exhausted
`retry_development` (3/3 — QA's model degenerated, see §8) and escalated via a real
`interrupt()` to `human_review` — the *designed* terminal state for exhausted retries.
The web API reported `status: "complete"` instead of `"awaiting_human"`. The Compare tab
would have shown a false green result.

**Root cause:** `LangGraphBackend` recompiles the graph on every call
(`_compile_for_run(mode, None)`); without `AI_TEAM_LANGGRAPH_SQLITE_PATH` set,
`resolve_sqlite_checkpointer()` falls back to a throwaway `sqlite3.connect(":memory:")`
**per compile**. The web server compiles the graph at least twice per run — once to
stream events, once independently in `_langgraph_hitl_status` to read back state for HITL
detection — so the status-check compile had its own empty checkpoint store and could
never see the run's actual interrupt.

**Fix** (`src/ai_team/ui/web/server.py`): default `AI_TEAM_LANGGRAPH_SQLITE_PATH` to
`workspace/.langgraph_checkpoints.sqlite` at server import time (`os.environ.setdefault`
— an explicit env var still wins), so every compile in the process shares one persistent
store.

**Verified**: a real `interrupt()` written by one compile is now visible via
`snap.tasks[].interrupts` to a second, independently constructed graph pointed at the
same file. Also fixed a pre-existing test-env leak
(`test_checkpointer_resolution.py` used a raw `os.environ.pop` instead of `monkeypatch`,
which — combined with §7 below — permanently cleared this var for the rest of the test
session).

### 7. Root cause of `workspace/tid/`: a global env-var leak, not just a stray test artifact

The dozens of stray UUID subdirectories under `workspace/tid/` (flagged at the end of the
prior turn as "out of scope, flagged separately") turned out to be a **real production
bug**, not test debris to just delete. All three backends (langgraph, claude-agent-sdk,
crewai) plus `main_flow.py` set `os.environ["PROJECT_WORKSPACE_DIR"]` as a **bare,
permanent mutation** — no restore — to scope the active per-run workspace for the global
`Settings` singleton. A LangGraph test called `iter_stream_events(thread_id="tid")` and
never got its override undone, so every later call in that pytest process that didn't set
its own override fell back to `./workspace/tid`.

This isn't only a test-hygiene issue: the identical bare-write pattern in production means
a long-lived process — the web server, or the Compare tab running three backends
together — can leak one run's workspace path into another's, corrupting workspace
isolation between concurrent/sequential runs. This is a more severe finding than the
original "flag separately" framing suggested.

**Fix**: new `ai_team.config.settings.scoped_workspace_dir()` context manager — sets
`PROJECT_WORKSPACE_DIR` + `reload_settings()` on entry, restores the prior value (or
unsets it) + reloads again on exit, via `try/finally` so it survives early returns,
generator `.close()`, and exceptions (including the `SystemExit` from
`main_flow.kickoff()`'s cost-decline path). Applied at all 6 call sites. Methods with
multi-`yield` generator bodies (LangGraph's `iter_stream_events`, Claude SDK's `stream()`)
were split into a thin public wrapper (enters the scope) delegating to a private `_impl`
generator, rather than reindenting the whole body under a `with`.

**Verified**: 18 new tests (`scoped_workspace_dir` set/restore/unset/exception/nesting;
fixed the leaking test to use `monkeypatch`); reran the exact leak repro
(`test_langgraph_results_bundle.py` alongside an assertion that `PROJECT_WORKSPACE_DIR`
is never `./workspace/tid` afterward) — no longer reproduces. Stray directory removed.

### 8. LangGraph's QA degeneration on deepseek — implemented 06-28's re-prompt lever

**Symptom, in the actual comparison run:** with §5-§7 fixed, LangGraph's testing phase
still failed — not from a guardrail bug this time, but because deepseek's QA agent
**degenerated into repeating "the test." hundreds of times**, never called `file_writer`,
and pytest collected zero tests (exit 5). This is the exact "agentic, not code" failure
class 06-28 documented — the model's fault, not the orchestration's — and salvage
(extracting fenced code from prose) can't help because the model never generated any code
to extract.

**What 06-28 proposed but didn't build:** lever #3, *"detect 'agent emitted no tool call'
within the loop and re-prompt immediately — cheaper than a full phase retry."*

**Implemented this session** (`subgraph_runners.py::testing_subgraph_node`): after the
quality gate reports `test_results.no_tests_collected` and salvage didn't recover a test
file, the node re-invokes the same testing subgraph **exactly once** with a blunt,
specific corrective message ("your last response did not call `file_writer`... the next
message must be a `file_writer` tool call"), then re-runs salvage + the gate. Falls back
cleanly to the original failing result if the re-prompt call itself raises, so the
graph-level `retry_development` loop still gets a clear signal. Bounded to one extra
attempt — does not become its own unbounded loop, and is far cheaper than the
full dev+test cycle `retry_development` would otherwise trigger.

**Verified**: 4 new tests (recovers via re-prompt; skips re-prompting when the first try
already passed or salvage already recovered tests; falls back safely when the re-prompt
call raises). Not yet re-verified against a live degenerating model in this session (the
final comparison run had already exhausted its graph-level retries and escalated to
`human_review` before this fix landed) — next real run is the verification.

### 9. CrewAI: the "never registers a run" verdict was partly a false reading — corrected

Four attempts earlier in this session showed CrewAI registering
`crewai_spend_guard_registered` and then **nothing else, ever** — no run entry in
`/api/runs`, no workspace directory. Read at the time as reconfirming 06-28's known-red
verdict. **That reading was wrong for three of the four.**

**Actual cause:** the server process alive during those attempts was **stale** — started
before the `scoped_workspace_dir` fix (commit `e300a86`, this session) landed. Directly
re-testing CrewAI against that same stale process reproduced it exactly:
`run_started` fired, then `{"type":"error","message":"cannot import name
'scoped_workspace_dir' from 'ai_team.config.settings'"}` — a crash in **this session's
own new code**, before CrewAI's flow ever ran. That import error, not a CrewAI defect,
is what killed most of this session's "CrewAI never registers" observations.

**What a genuinely fresh server showed:** restarted clean, re-ran the same brief.
CrewAI progressed for real — cost confirm → intake → planning (product_owner *and*
architect LLM calls both completed) → development → a legitimate guardrail failure
("Frontend implementation must produce at least one CodeFile", same class as LangGraph's
dev-phase guardrail failures today) → `retry_backoff attempt=2 delay_seconds=4.0`.

**Then it hung.** After that backoff log line, ~10 minutes passed with the process
pinned at 80%+ CPU and **zero further log output** — no retry, no next LLM call, no
error, nothing but health-check polling from the outside. This *does* match 06-28's
documented signature (Rich-console recursion / thread deadlock inside the retry path,
ignoring its own timeout) — just triggered later in the pipeline (mid-retry-backoff)
than previously observed (immediately after spend-guard registration).

**Corrected verdict:**
- CrewAI **can** start, run real LLM calls, and progress through intake → planning →
  development on the web/Compare path — this session's "never even registers" framing
  was too strong and was mostly a self-inflicted testing artifact (stale server).
- CrewAI **does still hang** for real, deterministically, inside its retry-recovery
  path once a dev-phase guardrail failure triggers a retry — consistent with 06-28's
  root cause (CrewAI's Rich console / event-bus fighting headless, retryable
  orchestration), just observed at a different trigger point than before.
- **Comparison-only status stands**, but the reason is now more precise: it's not that
  CrewAI can't run at all, it's that it can't survive its own retry path without
  deadlocking. 06-28's "bounded last attempt" list (hard subprocess-level kill wrapper
  so the wall-clock timeout actually reaps it; "tests exist on disk → don't
  retry_development" short-circuit to avoid entering the retry path at all) is still the
  right next move if this gets revisited — not further whack-a-mole on the import/startup
  side, which is now fixed.

**Process note for future sessions:** always verify the web server was restarted *after*
your most recent `src/` commit before drawing conclusions from a "silent" run — a stale
process swallowing a new `ImportError` inside a broad `except Exception` looks
identical to "the backend won't even start."

### 10. Run + comparison history persisted to SQLite (closes the gap §9 exposed)

**Motivation:** §9's stale-server confusion was hard to diagnose largely because there
was nowhere to look — `RunState` tracked runs in an in-memory dict only, wiped on every
restart, and the Compare tab's three backend runs (`crewai`/`langgraph`/`claude-agent-sdk`)
had no server-side link between them at all (three fully independent `/ws/run`
connections). Asked directly "do we have all the comparison data?" the honest answer was
no — Claude SDK's and LangGraph's data came from one Compare attempt, CrewAI's real
(non-crashed) data came from a *different*, smaller solo test run afterward, and none of
it would have survived another restart.

**Fix** (`src/ai_team/core/run_store.py`, new): `RunStore` mirrors every run lifecycle
event (`create_run` → `upsert_run`, status changes → `update_status`) into
`data/memory.db` — the same SQLite file `ai_team.memory.memory_config.LongTermStore`
uses, but independent of `MemorySettings.memory_enabled` (run history should persist
regardless of whether agent long-term memory is on). All writes are best-effort — a
persistence failure logs a warning and never breaks a live run.

The frontend's Compare tab now generates one `comparison_id` (`crypto.randomUUID()`) per
"Run All Backends" click and passes it to all three `startRun()` calls; the server
records a `comparison_runs` link table so the three backend runs can be looked up
together after the fact, even across restarts.

New endpoints: `GET /api/runs/history` (persisted, survives restarts — unlike the
existing in-memory `GET /api/runs`), `GET /api/comparisons` (recent Compare sessions),
`GET /api/comparisons/{comparison_id}` (one comparison's linked runs).

**Verified live, including the exact failure mode this was built to fix:** launched a run
via a raw websocket client with a `comparison_id`, confirmed it in both new endpoints,
then killed the server and restarted it — `GET /api/runs` (in-memory) was correctly
empty, `GET /api/runs/history` (persisted) still had the run. Also caught a real bug
during that live test: `/api/runs/history` was registered *after* `/api/runs/{run_id}` in
FastAPI's route table, so `"history"` was being swallowed as an unknown `run_id` and
404'ing — fixed the registration order. Unit tests alone would not have caught this
(route-ordering bugs only show up when you actually hit the HTTP layer).

18 new tests (`RunStore` CRUD, comparison grouping, persistence-failure isolation) plus
an autouse test fixture (`tests/unit/ui/conftest.py`) that routes every `RunStore`
construction — including fresh `RunState()` instances some existing tests build
directly, not just the module-level `state` — to an in-memory DB, so the test suite
never writes into the real `data/memory.db`. Caught and fixed a real instance of this
during development: an earlier, unpatched test run had left 27 stale rows in the real DB.

## Dev servers kept dying mid-run (environmental, worked around)

Both `uv run ai-team-web` and the Vite frontend dev server were killed by session
teardown (`/model` switch, background-task-timeout notifications, port conflicts from
stale duplicate processes) **repeatedly** this session, each time interrupting an
in-flight comparison run mid-phase. Also found: **three stale `ai-team-web` processes
running simultaneously** at one point, each with its own in-memory run registry — which
made a genuinely-running LangGraph run vanish from `/api/runs` (it was on a different
process than the one being queried).

**Fix applied**: started both servers with `nohup ... &` + `disown` instead of the
harness's tracked background-task mechanism, so they survive session/model-switch
teardown. Always check `ps aux | grep ai-team-web | grep -v grep` before trusting
`/api/runs` — kill all matches and restart exactly one if more than a pair (uv wrapper +
python child) show up.
