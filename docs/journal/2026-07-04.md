## Bug fixes (all live-verified)

| # | Problem | Root cause | Fix |
|---|---------|-----------|-----|
| 1 | Compare tab launched 3 backends in parallel on a **fresh** workspace and all got the same `run_id` | `allocate_run_id`'s mkdir-based reservation was skipped whenever none of the search roots existed yet — the fallback path had no reservation at all | Create the first search root before reserving, not only when it already exists (`d688002`) |
| 2 | CrewAI runs nested under a stale sibling run's directory in a long-lived process (web server) | `CrewAIBackend.run()` fell back to `nullcontext()`/`get_settings()` when no `workspace_dir` override was given, inheriting whatever `PROJECT_WORKSPACE_DIR` an earlier run left in `os.environ` | Always scope explicitly to the literal `"./workspace"` instead of trusting env state (`43d66d2`) |
| 3 | `ResultsBundle` double-nested `workspace/<id>/<id>/` when constructed from inside an already-scoped path | Blindly appended `project_id` to `workspace_dir` without checking if it was already there | Check whether `workspace_dir` already ends in `project_id` before appending (`43d66d2`) |
| 4 | CrewAI's Compare column showed permanent zeros — no live monitor, subprocess-isolated | No code ever touched the `TeamMonitor` for CrewAI's `run_started`/`run_finished` events | Set phase to `DEVELOPMENT` on start; backfill files/tasks/tests from the subprocess result payload on finish via new `_apply_crewai_result_to_monitor()` (`2388bca`) |
| 5 | Compare run state vanished on page reload even though the server was still tracking the run | React state only, nothing persisted client-side | `localStorage` stores `comparison_id`+`run_ids`; reattach via `GET /comparisons/{id}` on mount (`4ce7dce`) |
| 6 | `run.json` `completed_at` stayed `null` forever; `logs/costs.jsonl` was always empty | No code path ever wrote either field after a run ended | New `ResultsBundle.finalize()` stamps completion + appends a spend row; wired into the server's `finish_run()` for every backend (`887549c`) |
| 7 | Claude SDK runs were invisible to the disk registry — no `output/runs/<id>/` at all | SDK backend never constructed a `ResultsBundle`; results lived only in the workspace | `_write_results_bundle()` called on both `run()` and `stream()` paths (`887549c`) |
| 8 | A run resumed through human review reported plain `complete`, indistinguishable from a run that actually passed its quality gate | No status distinction existed between "passed" and "operator overrode" | `approved_via_hitl` flag → `final_status: complete_approved` (`887549c`) |
| 9 | Every comparison table in the project was built from a single run, and single runs disagreed with themselves within the hour (CrewAI 6m50s → 10m41s, LangGraph clean → HITL-escalation, same config) | No tooling existed to run N times and aggregate | `scripts/run_smoke_batch.py` — runs N per backend via `run_demo.py`, reads tests/spend back from the (now-working) bundles, emits a variance table |
| 10 | Batch runner's first shakedown run failed LangGraph's lint gate with "Command not found: ruff" | Detached background shell didn't have `.venv/bin` on `PATH` | Prepend `.venv/bin` to the subprocess env explicitly |
| 11 | Batch runner misread SDK test results as "no data" | A **third** `test_results` schema exists — SDK nests counts under `summary`, CrewAI uses flat counts, LangGraph uses `{'passed': bool, 'tests': {...}}` | Normalize all three in `_tests_from_state()` |
| 12 | Screenshot tooling: could not get any browser screenshot onto disk where Bash could see it | Three separate tool sandboxes (preview browser, Playwright-driven Chrome, computer-use MCP) each render/save invisibly to the others — confirmed by `find`-ing the whole home dir for a "saved" file and finding nothing | Killed the preview-managed dev servers, restarted as plain background processes, drove a **real** Chrome window via the claude-in-chrome extension, then used plain macOS `screencapture -R` against that window's on-screen bounds (from `osascript ... get bounds of window 1`) — OS-level capture has no sandbox to cross |
