## Top priority: crewai retry loop

### Root cause
`retry_development` listener at `src/ai_team/flows/main_flow.py:1116` emits multiple times (8× at same timestamp seen in logs). Each emission triggers `run_development` → tests fail → `route_after_testing` returns `"retry_development"` → loop. `state.can_retry(ProjectPhase.TESTING)` is checked but development retry is not bounded independently.

### Investigation path
1. Check `src/ai_team/flows/routing.py:204` (`route_after_testing`) — does it increment a dev retry counter?
2. Check if crewai event bus double-emits `retry_development` on certain error conditions
3. Check `src/ai_team/flows/main_flow.py` around `handle_development_error` / `route_after_development_error` — may be emitting `retry_development` in addition to normal routing

### Suggested fix
Add a guard in `retry_development` listener: check `consecutive_failures_development` in state metadata, bail out (emit `escalate_to_human`) if > 3.

## Second priority: watchdog per-backend stall

Current watchdog (`/tmp/eval_watchdog.sh`) tracks a single "last line" from `[compare]`. When crewai judge stalls but SDK heartbeat keeps changing, stall timer resets — watchdog never kills the hung judge.

Fix: track stall timestamp per backend name, kill only the stalled backend's subprocess.
