## Remaining issues

### crewai: `retrying_development` infinite loop (~50% of runs)
`retry_development` listener emits multiple times at same timestamp ‚Äî crewai flow event bus bug. `route_after_testing` routes to `retry_development`, which fires `run_development`, which fails tests, which routes back to `retry_development` ‚Äî no counter checked. `state.can_retry(ProjectPhase.TESTING)` is called but development retry count not bounded separately.

**Investigation starting point:** `src/ai_team/flows/main_flow.py:1116` (`retry_development` listener), `src/ai_team/flows/routing.py:204` (`route_after_testing`). The 8√ó duplicate log lines at same timestamp suggest crewai event bus double-emitting.

**Workaround:** run crewai eval separately: `uv run python -m evals.run_evals --backend crewai --no-judge` and re-run if stuck.

### LLM judge: Anthropic API slow responses
Judge hangs 60-120s per criterion on slow API days. `timeout=float(60)` set but responses sometimes take 90-120s (valid slow response, not a bug). Watchdog auto-kills at 120s stall after `phase=complete`.

**Watchdog bug:** stall detection resets if any backend's heartbeat line changes ‚Äî can't distinguish per-backend stalls. Fix: track stall per-backend in watchdog.

### crewai: QA crew runs many iterations
QA crew iterates 9+ times on coverage analysis for a trivial `add()` function. Slow (adds 200-300s). Consider adding iteration limit to QA crew config.
